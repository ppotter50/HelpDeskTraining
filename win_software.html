Autodesk Fusion 360

  • AutoDesk Fusion 360 virtualization can be very slow. This is because the virtualization installs this application locally instead of actually virtualizing. (This should not be public information, internal only.)
  • When the application virtualizes it first installs the software to the computer, then installs any available updates, then posts a shortcut to the software on the desktop.
    • These processes all check to ensure they have not already been done (i.e., it won't install the software twice).
    • When these changes are made, they are applied directly to the computer and maintained on a computer-to-computer basis, not on a user basis.
      • This means that when the user logs off, the software will remain installed for the next user and virtualization for them will be much faster.
    • This can lead to really long virtualization times, specifically if:
      • The software hasn't been used since the last Windows release.
      • The software is old and not up-to-date.
      • There are a lot of people trying to update simultaneously (e.g., a whole lab class). Network load limits are to blame.
  • When a user logs off or devirtualizes the software, the shortcut is removed but the software remains installed.
    • This shortens the next user's virtualization times while preventing users from knowing that the software is locally installed on the machine.
    • We don't want them to know this due to license security concerns!!!
  • Remember: We do not tell users that the software is locally installed, we only tell them that we know things are slow, and that this is to be expected.
